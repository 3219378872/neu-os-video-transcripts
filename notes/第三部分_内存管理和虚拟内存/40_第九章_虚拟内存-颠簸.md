# 第九章 虚拟内存——颠簸

## 本讲主题

介绍颠簸的概念、产生原因、恶性循环机制及基于局部性原理的解决方法。

## 核心知识点

- **颠簸的定义**
  - 多个进程频繁进行页面调入调出，而非执行实际计算的现象
  - 本质原因：物理内存不足以容纳所有并发进程的活跃页
  - 颠簸发生时系统处于崩溃边缘，几乎无法正常运行
- **颠簸的恶性循环（五步连锁）**
  - 第一步：进程 A 帧不够 → 全局置换抢占进程 B 的帧
  - 第二步：进程 B 帧被抢 → 又去抢进程 C 的帧，雪球越滚越大
  - 第三步：多个进程都在磁盘等待队列中排队等待调页，无实际计算
  - 第四步：CPU 利用率显著下降 → 操作系统误判多道程序度过低
  - 第五步：操作系统引入更多新进程 → 内存更加不足 → 系统崩溃
- **颠簸的判断条件**
  - 所有并发进程的局部（活跃页集合）之和 > 物理内存总量 → 必然颠簸
  - 所有并发进程的局部之和 < 物理内存总量 → 不会颠簸
- **解决方法**
  - 基于局部性原理（Locality Principle）：进程在某时间段内集中访问一组页
  - 工作集合模型（Working Set Model）：计算每个进程当前活跃的页集合大小
  - 判断公式：∑ 工作集大小 vs 物理内存总量
  - 若所有进程的工作集之和超过物理内存 → 降低多道程序度（减少并发进程数）
  - 示例：10 个进程并发运行，计算各进程局部之和是否超过物理内存容量

## 重要对比/关系

| 系统状态 | 工作集之和 vs 物理内存 | CPU 利用率 | 处理措施 |
|----------|----------------------|-----------|----------|
| 正常运行 | 工作集之和 < 物理内存 | 正常 | 可增加多道程序度 |
| 颠簸状态 | 工作集之和 > 物理内存 | 骤降 | 降低多道程序度 |

## 关键术语

| 术语 | 英文 | 定义 |
|------|------|------|
| 颠簸 | Thrashing | 进程频繁调页而非执行计算的现象 |
| 多道程序度 | Degree of Multiprogramming | 系统中并发运行的进程数量 |
| 局部性原理 | Locality Principle | 进程在某时段内集中访问一组页的特性 |
| 工作集合 | Working Set | 进程当前活跃使用的页的集合 |
| CPU 利用率 | CPU Utilization | CPU 实际执行计算的时间比例 |

## 要点总结

1. **颠簸**是多个进程频繁调页导致 CPU 利用率骤降的严重现象
2. 颠簸的恶性循环：帧不足→互相抢占→CPU 空闲→**操作系统误加新进程**→崩溃
3. 判断标准：所有进程**工作集之和**是否超过物理内存总量
4. 解决颠簸的核心方法是基于**局部性原理**，使用工作集合模型
5. 当工作集总和超过物理内存时，需**降低多道程序度**以避免颠簸
