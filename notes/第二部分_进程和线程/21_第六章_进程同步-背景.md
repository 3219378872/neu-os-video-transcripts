# 第六章 进程同步——背景

## 本讲主题

介绍进程同步的背景，通过改进的生产者-消费者问题引出竞争条件（Race Condition）问题。

## 核心知识点

- **进程同步的必要性**
  - 协作进程对共享数据的并发访问容易产生数据不一致
  - 需要规定访问顺序以保证数据一致性
- **改进的生产者-消费者问题**
  - 引入 counter 共享变量记录缓冲区中已使用的数据项数
  - counter 初始化为 0
  - 生产者：缓冲区满（counter == BUFFER_SIZE）时等待；生产后 counter++
  - 消费者：缓冲区空（counter == 0）时等待；消费后 counter--
  - 优势：可利用全部 N 个缓冲区（不再是 N-1 个）
- **counter++ 和 counter-- 的底层实现**
  - counter++ 的三条机器指令：
    1. register1 = counter
    2. register1 = register1 + 1
    3. counter = register1
  - counter-- 类似，使用 register2
- **竞争条件（Race Condition）**
  - 并发执行 counter++ 和 counter-- 时，如果在中间发生上下文切换：
    - 假设 counter = 5
    - 生产者执行前两步：register1 = 6
    - 上下文切换，消费者执行前两步：register2 = 4
    - 最终 counter 可能是 4 或 6，取决于最后执行的赋值语句
  - **定义**：多个进程并发访问共享数据，最终值取决于特定的执行顺序
  - 这种不确定性是进程同步要解决的核心问题

## 关键术语

| 术语 | 英文 | 定义 |
|------|------|------|
| 进程同步 | Process Synchronization | 控制协作进程对共享数据的访问顺序 |
| 竞争条件 | Race Condition | 共享数据的最终值取决于进程的执行顺序 |
| 数据一致性 | Data Consistency | 共享数据在并发访问下保持正确 |
| 共享变量 | Shared Variable | 多个进程共同访问的变量 |
| 原子操作 | Atomic Operation | 不可被中断的操作 |

## 要点总结

1. 协作进程并发访问共享数据时，容易产生**数据不一致**问题
2. counter++ 和 counter-- 在底层各由**三条机器指令**实现，并非原子操作
3. 中间发生上下文切换会导致**竞争条件**，共享变量的最终值不可预测
4. 竞争条件的本质：共享数据的最终值取决于**特定的执行顺序**
5. 解决竞争条件需要对协作进程进行**同步**，这是后续临界区问题的出发点
