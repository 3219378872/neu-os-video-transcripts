# 第四章 进程通信

## 本讲主题

介绍进程间通信（IPC）的消息传递系统，包括直接通信与间接通信、阻塞与非阻塞通信方式。

## 核心知识点

- **消息传递系统（MPS）**
  - 操作系统提供两种原语：send（发送）和 receive（接收）
  - 需要考虑的设计问题：链接建立方式、连接进程数、链路数量、承载能力、消息长度（定长/变长）、通信方向（单向/双向）
- **直接通信**
  - **对称通信**：send(P, message) 和 receive(Q, message) 都需明确指定对方进程 ID
  - **非对称通信**：send 需指定接收者 ID，receive 不指定发送者 ID（接受任意发送者）
  - 性质：链接自动建立、一对一、一个链路、可单向或双向
  - 缺点：进程 ID 变更时需修改所有相关 send/receive 语句，限制模块化设计
- **间接通信（邮箱通信）**
  - 通过操作系统维护的邮箱（Mailbox）中转消息
  - 每个邮箱有独立 ID，多个进程共享邮箱即可通信
  - send(A, message)、receive(A, message) 中 A 为邮箱 ID
  - 一个邮箱可对应多个进程，一对进程可共享多个邮箱
  - 多接收者冲突解决：限制邮箱只连接两个进程 / 只允许一个进程执行 receive / OS 随机选择并通知发送者
- **管道通信（Pipe）**
  - 第三种 IPC 方式，通过共享文件进行通信，不太常用
- **阻塞与非阻塞**
  - **阻塞发送（同步）**：发送后进入等待，直到接收方确认
  - **非阻塞发送（异步）**：发送后继续执行，不等待确认
  - **阻塞接收**：接收方等待消息到来后才继续
  - **非阻塞接收**：接收方继续执行，消息到来时通过中断通知

## 关键术语

| 术语 | 英文 | 定义 |
|------|------|------|
| 消息传递 | Message Passing | 通过 OS 消息队列进行进程间通信 |
| 直接通信 | Direct Communication | 通信时需明确指定对方进程 ID |
| 间接通信 | Indirect Communication | 通过邮箱中转消息的通信方式 |
| 邮箱 | Mailbox | OS 维护的消息中转数据结构 |
| 阻塞发送 | Blocking Send | 发送后等待接收方确认的同步方式 |
| 非阻塞发送 | Non-blocking Send | 发送后不等待确认的异步方式 |
| 管道 | Pipe | 通过共享文件进行进程间通信 |

## 重要对比/关系

| 对比项 | 直接通信 | 间接通信 |
|--------|---------|---------|
| 中介 | 无，直接指定进程 ID | 通过邮箱中转 |
| 链路数 | 一对一 | 多对多 |
| 灵活性 | 低（ID 变更需改代码） | 高（通过邮箱解耦） |

## 要点总结

1. 消息传递系统提供 **send** 和 **receive** 两种原语实现进程间通信
2. 直接通信需指定进程 ID，分为对称和非对称两种；缺点是**耦合度高**
3. 间接通信通过**邮箱**解耦，多个进程共享邮箱即可通信，灵活性更高
4. 阻塞/非阻塞通信影响发送方和接收方的等待行为，对应**同步/异步**模式
5. IPC 主要方式：共享内存、消息传递、管道通信
