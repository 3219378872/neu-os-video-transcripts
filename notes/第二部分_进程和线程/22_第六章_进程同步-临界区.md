# 第六章 进程同步——临界区

## 本讲主题

介绍临界区问题的定义、三个必须满足的条件（互斥、前进、有限等待），以及 Peterson 算法的实现。

## 核心知识点

- **临界区问题（Critical Section Problem）**
  - N 个协作进程共享数据，每个进程中操作共享数据的代码段称为**临界区**
  - 核心要求：对共享数据的访问必须互斥
  - 解决方案结构：进入区 → 临界区 → 离开区 → 剩余区
- **三个必要条件**
  - **互斥（Mutual Exclusion）**：一个进程在临界区时，其他进程不能进入
  - **前进（Progress）**：
    - 第一层：一个进程想进入临界区且无其他进程在临界区时，应允许进入
    - 第二层：刚从临界区出来的进程不能立即再次进入（让给其他等待者）
  - **有限等待（Bounded Waiting）**：从声明进入到实际进入之间，其他进程进入次数有上限
- **Peterson 算法**
  - 适用于两个进程 Pi 和 Pj 的同步
  - 共享变量：
    - `turn`：决定哪个进程可以进入临界区（turn=i 则 Pi 可进入）
    - `flag[i/j]`：声明是否想进入临界区（true=想进入）
  - Pi 的进入区代码：
    1. `flag[i] = true`（声明想进入）
    2. `turn = j`（谦让：把机会给对方）
    3. `while(flag[j] && turn == j);`（等待：对方想进且轮到对方时空循环）
  - Pi 的离开区代码：`flag[i] = false`
  - **满足三条件的验证**：
    - 互斥：由 turn 变量保证，一次只有一个进程进入
    - 前进（第一层）：对方 flag 为 false 时直接跳过 while
    - 前进（第二层）：刚出来的进程再次进入时 turn=对方，让出机会
    - 有限等待：等待次数为 1

## 关键术语

| 术语 | 英文 | 定义 |
|------|------|------|
| 临界区 | Critical Section | 进程中操作共享数据的代码段 |
| 互斥 | Mutual Exclusion | 一次只允许一个进程在临界区 |
| 前进 | Progress | 不阻碍其他进程进入临界区 |
| 有限等待 | Bounded Waiting | 等待进入的次数有上限 |
| Peterson 算法 | Peterson's Algorithm | 用 turn 和 flag 变量解决两进程临界区问题 |

## 要点总结

1. 临界区问题的解决方案必须同时满足**互斥、前进、有限等待**三个条件
2. Peterson 算法用 **turn**（决定权）和 **flag**（意愿声明）两个共享变量实现同步
3. `turn = j` 体现**谦让精神**：把进入机会让给对方，是保证前进条件第二层的关键
4. Peterson 算法的有限等待次数为 **1**，即最多等对方进入一次
5. 进入区和离开区的代码设计直接决定同步方案是否正确
